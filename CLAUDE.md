# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```
make fixtures-p<N>   # generate fixtures for problem N (runs scripts/gen_p<N>.py)
make test-p<N>       # build and test problem N against fixtures
```

The test runner (`scripts/test.py`) copies `submissions/p<N>.hs` → `p<N>/Solution.hs`, builds with `stack build haskell-leetcode:exe:p<N>`, then runs each fixture through the binary and diffs stdout against the expected output. For JSON outputs (lists), comparison is order-insensitive.

## Architecture

Each problem lives in three places:

1. **`problems/p<N>/`** — the runnable executable. `Main.hs` wires up I/O; `Solution.hs` is generated by the test harness from the submission (do not hand-edit it).
2. **`submissions/p<N>.hs`** — the authoritative solution file. This is the only file you need to edit when working on a problem. It must export exactly the function(s) that `problems/p<N>/Main.hs` imports.
3. **`fixtures/p<N>/`** — `NN.in` / `NN.out` pairs, generated by `scripts/gen_p<N>.py`. The generator is an oracle (ground-truth Python implementation of the same problem) that exhaustively covers the valid input domain.

Adding a new problem requires:
- `problems/p<N>/Main.hs` — parse stdin, call the solution function, print result
- `problems/p<N>/Solution.hs` — placeholder (will be overwritten by the harness)
- `submissions/p<N>.hs` — the real solution
- `scripts/gen_p<N>.py` — oracle + fixture generator
- A new executable stanza in `package.yaml` with `source-dirs: problems/p<N>`

## Problem selection

`leetcode-small-testcase-domains.md` catalogs LeetCode problems whose complete valid input domain is small enough to enumerate exhaustively. Prefer problems from that list: the generator can cover every legal input and the solution can be verified for correctness rather than just a sample of cases.

## Stack / build

`stack.yaml` pins the resolver. All dependencies are `base`-only by default; add extras to the relevant executable stanza in `package.yaml` when needed.
